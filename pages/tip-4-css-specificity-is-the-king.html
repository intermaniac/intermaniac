Tip #4: 💄 CSS specificity is the king! 👑 If you ever wonder why one style is applied but not the other, it’s time to go back and make sure CSS specifics holds no secrets for you! More about this in the 🧵👇🏼 #CSS #FrontEndDeveloper https://t.co/NH2uxedRF1

1️⃣ What CSS specificity is? Styles can have multiple competing rules and try to change background-color of the same element multiple times to different value. Browser needs to make a call at some point thus specificity! Rules with highest specificity will get priority! https://t.co/TlEApj9z3R

It means that if two competing rules are important! then browser will take a look at selectors. If that is still not enough (same specificity) final decision will be made base on the document order (now we also have @layers but that’s a separate topic 🤓).

2️⃣ How specificity is calculated? There are 3 main levels at which browser calculates specificity: - !important - inline/selectors - document order !important is most important, so it takes priority over inline/selectors and inline/selectors takes priority over document order. https://t.co/LRBXkI6WDx

3️⃣ How inline/selector’s specificity is calculated? Now this is where most confusion comes from but at core it’s exactly the same principle: selectors are divided into more and less important, if one rule has more high priority selectors than the others, it wins 🏆

- Inlines are checked - inline rule wins over non inline - Then ID selectors are counted - rule with more wins - Then classes and attributes selectors are counted - rule with more wins - Then element selectors are counted - rule with more wins

You can visualize this by 4 counters and assign proper value base on number of ID, classes, attributes and elements selectors for a given rule then compare it with another one - bigger number wins! ie. 0130 vs 0200 ⇒ 200 🥇 https://t.co/BMIweBPmSe

If that is still not enough, document order decides on which rule should take precedence: - rule defined later in the same css file take precedence - if rules are defined in separate css files order in which these files where defined in html file decides (later wins).

If there is one thing to remember from this thread, please take this: Always try to write as unspecific rules as possible so you wont take other tools out of your tool belt.

Imagine having in-line style - from now on only way to override it is !important. Once that’s done you’re left with inline !importnat 🤯

SASS, Tailwind CSS, Windi CSS, CSS Modules, ShadowDom, static css, generated css or whatever works for you, it doesn't matter - these rules work in every scenario 💪🏼
