Bigger cache would mean higher cost

How to put more in cache without additional cost? Firecracker is more efficient - MicroVM

Firecracker can run in a fraction of a second with the same isolation as VM

Firecracker is optimized to shutdown and restart VMs in a fraction of a second rather than the t2 instances - allows to keep function running longer

With Firecracker, the cold start latency rate was reduced by 50%. Even with more spiky workloads that it attracted

Problem with caches: no matter the size, eventually you have a cache miss

So: need to make cold starts faster

Cold start lifecycle: VM from common snapshot. Transfer customer function code - time depends on size / dependencies. Once loaded: initialization https://t.co/UviBlXyT0r

In Java, initialization is particularly long. Imagine taking a snapshot after initialization and starting with this snapshot https://t.co/BGlU7nnt46

Announcing: AWS Lambda SnapStart https://t.co/09EPwz0hPp

SnapStart is available tonight at no additional cost

Up to 90% reduction in cold start latency - virtually indistinguishable from cache hit

Innovation: Enabling uniqueness - required for pseudo-random number generators. If generated during initialization, it would be reused and not be random. To avoid this: need to find critical sections of code to ensure random seeds are calculated _after_ snapshot is restored

Guidance available for custom crypto libraries

Innovation #2: dealing with snapshots efficiently. Storing all functions would be expensive. Only part that is different: function part. So there can be a base snapshot https://t.co/3unkIIBTgb

Innovation #3: boot with minimal latency: predictive snapshot loading. Don't need to download full snapshot to start. Load the rest while it is booting. VM, unlike video, jumps all over the place. But jumping around is predictable.

Performance work is never done

Many more sessions on performance, diving deeper and covers other topics https://t.co/ZPwpPYPYeP
