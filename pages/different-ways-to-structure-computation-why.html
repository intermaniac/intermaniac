A thread on different ways to structure computation /why eth is decentralized google sheets, not decentralized AWS (coming from a day-post-covid-booster brain, so apologies if it's less than coherent)

Basically, large computation platforms can be placed along the spectrum of Computation vs State -- some platforms have state as a first class citizen and compute exists to manipulate state, others have vice versa (2/n)

As an example of a state-first platform: a database or a spreadsheet. These are tools primarily intended to keep track of information, and while they do have computational sides (stored procedures, formulas, etc), those computational tasks exist to manipulate state (3/n)

As an example of a computational-first platform: a calculator. Calculators exist to, well, calculate, and while they can store state in the form of prior results, that state basically only exists to aid in computing future results (4/n)

Eth (and most other blockchains), in this ontology, is a state-first distributed system -- you don't run most smart contracts because you care about their _results_, you do so because you care about how those contracts _change the state of the chain_ while executing (5/n)

In contrast, classic cloud providers like AWS EC2 are primarily about _the results of their computation_, and any stored state is used primarily to aid in the calculation of those results (6/n)

Now, why care about the distinction here? Basically, because if state is a first class citizen, you need to _make sure_ that state is kept accurately -- that means minimal parallelism, byzantine tolerance, etc (7/n)

In contrast, with computation as a first class citizen, you _can_ parallelize -- but your ability to have user-facing state available/up to date constantly is hampered. (8/n)

Before building a computation product, ask yourself: is what matters your state, or your results? Make tradeoffs accordingly. (9/9)
