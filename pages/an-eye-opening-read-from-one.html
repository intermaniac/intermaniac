An eye opening read from one of the maintainers of @emotioncss. All this time I was on the boat that runtime concerns of css-in-js libs are observably insignificant compared to dom updates and that style recalculation is extremely cheap. A thread. ðŸ§µ https://t.co/SJ8Lt5iGT9

> Frequently inserting CSS rules forces the browser to do a lot of extra work I didn't realize that style serialization during render comes at a noticeable cost. Serialization during render vs outside React's lifecycle has observable perf improvements.

> Runtime CSS-in-JS libraries work by inserting new style rules when components render, and this is bad for performance on a fundamental level. My takeaway here is to avoid writing styles within the component when those styles are not dependent on props.

@sebmarkbage explains it clearly in reactwg discussion https://t.co/qjXtFC1uzI > The worst thing you can do is insert them during React renders.

I like that the post ends up recommending css modules because I always found it to be the sweet spot between colocation and style composition if you're ready to compromise type safety.

I've been hearing good things about vanilla-extract and gotta give it a go sometime. Looks like it's the type safe evolution of css modules.

I still don't like the fact that we need to use a compiler of sorts for a good styling solution while building pages with React. But it's definitely better than the perf costs of runtime css which can easily get out of hand when left unchecked.
