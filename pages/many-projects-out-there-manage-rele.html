Many projects out there manage releases with git branches 🤔 or it is what they try to do 🤷‍♂️ hotfix, stable, and dev branches are still common but: it is an anti-pattern, and there is a better alternative here is why its wrong, and what to do instead a thread 🧵👇 1/11

i think every developer who has worked on: 🔸a large project 🔸with multiple branches has seen big merge conflicts. like merges with conflicts in 100+ files. if you've never seen one, consider yourself lucky 😅 2/11

why is the idea of having different branches for release and development so tempting? i think it is a case of developers overestimating what they are capable of. they estimate that bug fixes will only be small and isolated to a few files. 3/11

for this assumption to be accurate, the project must be mature and well architected. but if the codebase is tightly coupled, big bug fixes are created. bug fixes that affect files all over the project, and commits on the hotfix branch are suddenly complex and tangled. 4/11

with big and complex commits in the hotfix branch, it becomes hard to backport those bug fixes to the development branch. you put yourself into merge hell. 5/11

maintaining a multi-branch strategy is a significant effort, and it needs the discipline to do it right. if you have the discipline, it is better to capture the value that is created. once you open the next release branch, the work on the old stable branch is lost. 6/11

an alternative to a multi-branch strategy is: 🔸branch by abstraction 💁‍♂️it is a design pattern to keep different implementations in a single source tree. i have used this pattern successfully to replace legacy components in a project with modern implementations. 7/11

there are different ways to implement the "branch by abstraction" pattern, i do it like this: 1️⃣ i add an interface for new functionality 2️⃣ i refactor the existing code to go through the interface 3️⃣ i put my new implementation behind the interface 8/11

after this 👆 i have two implementations behind one interface, the new feature, and the old implementation. at this point, i can decide to enable the new code at any time. i can hold it back to keep the release stable, or i can ship it. or add a feature toggle for it. 9/11

using "branch by abstraction" is a great pattern: 🔸it removes the need for release and development branches 🔸it automatically adds structure to my code now i have to add an interface for new features, and i am forced to improve the structure of my code. 10/11

starting to use the "branch by abstraction" is really a win-win situation. you get a stable development branch, and you improve the structure of your code,... for free. 🔥 💪 11/11
