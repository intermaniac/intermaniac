Last month I helped a client scale their WordPress membership site to handle thousands of simultaneous logged-in users. The tricky part here is "logged-in users," because you can't use a typical caching plugin the way you would with a read-only site. Here's how I did it. ðŸ§µ

Some background: The client runs a *very* successful membership site using WordPress and Wishlist Member. And their latest membership drive was even more successful than past ones. Nothing but good news â€” until the live welcome call, when the site crumpled under the load.

So now we needed a fix â€” and fast. Not only was there another member call in a couple of days, but we needed to keep downtime to an absolute minimum during any upgrades. I began by reviewing the current hosting setup.

They were already running on solid hosting â€” a server from @digitalocean with 8 CPU cores and 16GB of RAM. The setup was optimized for WP with @serverpilot. And they were serving cached pages to non members (the public homepage / sales page). All good, but not enough.

The first change I made was upgrading the server, spinning up a new one with 32 CPU cores and 64GB of RAM. As far as hardware goes, this is a BEAST. Then I ran a load test with @loaderio. Better, but still not enough. I am undeterred.

Next I took a look at the database using some tools built into @wpcli. DBs tend to get cranky as they age and cruft accrues and best practices change. And sure enough, I saw that this DB was still using MyISAM as the storage engine on some tables. I converted these to InnoDB.

I also ran MySQLTuner on the server, and implemented all the database configuration changes that it suggested. Then I ran another load test. Again, not enough. In fact, no discernible difference. Time for Redis.

Redis is special kind of cache that keeps common DB queries in memory, so the server doesn't have fetch the same info from the disk over and over again. I worried this might cause problems, because logged-in users have to be served unique page info, but it worked just fine.

With Redis I finally saw a significant improvement in the load test, but it *still* wasn't enough. Fortunately I wasn't out of tricks yet.

Two final steps got me over the finish line: 1. Splitting the WP database off onto a server of its own. This was simple using @digitalocean and its preconfigured MySQL image. 2. Installing @wprocket and implementing it's specialized cache for logged-in users.

At the beginning of this thread I mentioned that most page caching systems cause problems for membership sites, because each user needs to be served some degree of unique info. But WP Rocket provides an option to create an individual cache for each logged-in user.

This means that each member is still being served unique info, but each page they visit is also cached specifically for them. It doesn't reduce server load as much as a single cache for all users, but it does help when members visit a page more than once per session.

The final load test passed with flying colors. The CPU never got above 12%. And later that night I migrated the live site to the new setup with only 15 minutes of downtime. Sweet, sweet victory.
