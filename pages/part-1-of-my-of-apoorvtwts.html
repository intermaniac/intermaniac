Part 1 of my thread of @apoorvtwts 's Git and GitHub masterclass #LearningWithApoorv

Git: Git is a version control system(VCS) for your files. What does Git do? - revert a selected file/ entire project back to prev state - compare changes - who last modified the files that's causing some issue. - who introduced an issue./when.

What is the meaning of version control? - Version control, also known as source control, is the practice of tracking and managing changes to software code. - Version control systems are software tools that help software teams manage changes to source code over time.

- Version control is a system that records changes to a file or set of files over time so that you can recall specific versions later.

Why do we need Version Control? - sometime we mess up things in project and we may want to go to prev version of the software where everything was working. VCS helps us to do this.

(continuing need of Version Control) - Initially developers used to maintain the version of their files manually. This is very error-proned because now there are so many files and codebases are large. That's where VCS came into the picture.

Types: * First VCS that was built was local VCS(concept was that they simply had a DB which maintained all the changes in a file) - It used to had the version DB that used to maintain what versions we have of the file.

- And it used to have the file. And it used to point to the current version of it. - This Version Control System used to store the data locally on the machine.

- The problem with local VCS was that it is helpful for maintaining personal projects. But in most of the projects, many developers work together so there it was a failure.

- That's where centralised VCS came into the picture. - It used to have single server or device somewhere in the world which used to contain all the versions for folks who used to work on a project.

- In this, there used to be centralised server accessible by all the developers and whatever changes a developer makes can be pulled by another developers.

- Downside of this -> - Single point of failure(means if there is a computer which is a server and if any issue happens let's say power cut or anything then others developers won't be able to access the server so any changes they'll push or pull won't happen

(continuing downside) - Hardware failure(storage)(let's say hard disk or SSD corrupts(unlikely) so it loses all the data and the entire project is gone).

Distributed VCS: - In centralised VCS dev going to have one version at a time(it can be any prev version or a current version) - In distributed VCS, server will have all the versions and dev will have clone of it so they'll have all the history

- so even if server is down or hardware fails, devs have all the history or versions and they can use it. - Git(not first VCS) is a distributed VCS.

How was Git created? Backstory: - Initially linux community used to give their updates in the form of archives/patches. - In 2002 they started using Bitkeeper(a distributed VCS). - In 2005, linux community had a fallout with Bitkeeper people.

- From there linux community created it's own distributed VCS. - It tried to solve few things-> 1. Speed 2. Simple design

(continuing...) 3. Strong support for non-linear development(non-linear means that development doesn't happen linearly like creating nav bar and then sections and all. It happens parallely. One dev may work on one feature and other on others).

(continuing...) 4. Fully distributed(for point 3 they want system to be fully distributed) 5. Efficiency with large projects. 6. That's when GIT was born
