A student asked me: "What AWS service should I use to collect and analyze click-stream events from the frontend?" üßµ

The tried and tested approach is to use: ‚Ä¢ Kinesis Firehose to ingest data, buffer and compress them, and optionally transform with lambda ‚Ä¢ S3 for data lake ‚Ä¢ Athena to analyze the data ‚Ä¢ QuickSight to visualize them in a dashboard Very common arch for BI workloads. https://t.co/vUcy6a3r4e

But, how do you get the data from the frontend to Kinesis Firehose? You can always put an API in front of the stream, and that's the common approach. But if all the API does is provide auth, then it's a waste because Kinesis Firehose provides IAM auth already.

So another way would be to use Cognito Identity Pool to issue temp AWS credentials to the frontend, based on a role that only allows publishing to the Kinesis Firehose stream. Now, the frontend can push events to the stream directly, using the temp IAM creds.

The great thing is that this works for both authed (after a user signs in) and unauthed use cases - for example, to track the progress of the sign-up funnel.

Another related question is "how can we collect backend-generated events in the same data lake?" You can always use AWS SDK to put events to the Firehose stream directly. But this is not great, it adds user-facing latency (for APIs) and complicates error handling.

I prefer to process them async via CloudWatch Logs. 1. Use a shared lib to publish events by writing BI events to stdout in a specific format. 2. Add CW Logs subscription filter for message format. 3. Forward matched messages to Lambda, which put events to the Firehose stream https://t.co/neDHjnr2E6

Now you have a solution that can be used in both Lambda and containers to record custom BI events. And the shared lib can expose a single function like this, and a filter pattern such as [timestamp, requestid, level="INFO", json="*\"level\":\"event\""] https://t.co/OEqoXgg4lv
