Not sure if Twitter is for useful things any more, or just shitposting, but hey why not. Here are 3 big lessons I learned while migrating 13 very large #MySQL databases last week. ðŸ§µ (This is not a "Big Data" post. My data isn't large enough to consider "Big Data".)

UP FRONT: I'm not a DBA. I'm just some schmuck who works on a small team where everyone wears all the hats (including DBA). This is probably all common knowledge to pro DBA's. But hey, these are lessons I had to learn the hard way last week.

Lesson #1: Disable constraints when doing bulk inserts. When importing 20mm rows, and you know the FK values exist b/c the FK's enforced integrity before you exported the data, the constraints are wasting a ton of time and cpu on import.

set foreign_key_checks = 0; /* import data here */ set foreign_key_checks = 1;

Lesson #2: Add multiple indexes with a single alter statement. Every alter causes a full table copy, then alters the copy, then drops the original and renames the copy. Don't repeat that on a 20mm row table for each of 5 indexes, add all 5 in one alter.

alter table Fubar add key 'ixFoo' ('foo'), add key 'ixBar' ('bar'), add key 'ixBaz' ('baz');

Lesson #3: mysqldump is not the only tool available to you for export. We're on AWS Aurora MySQL, which means we can use "select ... into outfile s3 's3://bucket/file.csv'" and then "load data from s3" to import into the new DB.

mysqldump is still great for handling all of the smaller tables. But for anything over 1gb we found it significantly faster (and worth the effort) to export to s3, create the table (with no indexes!), bulk import, then add the indexes.

BONUS TIP: if you have the ability, consider importing only a subset of the data at first. The export to S3 approach automatically splits the CSV's into 6gb files, so it was easy for us to import only the most recent data first.

This allowed us to get the app back online faster, and we simply posted a message to users that old data was still being loaded but that they could continue to use the app in the meantime. Then we queued up all of the old data imports and went to bed.

It was so pleasant to wake up to a notification that the import was complete and all was right with the world. Such a vast difference to the way the week started, working until 3am several nights to fight the data into the DB.

That's all of the tips I have for you, but if you want more details, notes on the things you need to be careful not to screw up, links to relevant docs, and more of the back story, then here's my article that has all of the above and more. https://t.co/bdA36uRXDA
