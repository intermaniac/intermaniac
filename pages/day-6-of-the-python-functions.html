Day 6 of the Python functions series: • more on *args & **kwargs In this thread, we'll look at functions with some combination of "normal" positional arguments, *args, and **kwargs Let's start with this example –> What do you think the output will be? <code in ALT text> /1 https://t.co/dnFxHTYCCu

The function definition has two parameters. The second one is a **kwargs as it has the double asterisk in front of it As we've seen in yesterday's thread, this means you can pass any number of keyword arguments which will be assigned to this parameter name /3

The function call has one positional argument, `red_team`, and three keyword arguments. You can have as many keyword arguments as you wish after the first required one /4

In the function body, `people` is a dictionary and therefore you can loop through it using the `items()` method `person` is a string containing the keyword you used in the function call /5

In this case it will be `"Stephen"` in the first iteration, `"Kate"` in the second, and `"Sharon"` in the third `points` will be the argument you pass which matches the keyword, so in this case it will be `10`, `8`, and `12` in the three iterations of the `for` loop /6

Here's the code with the output also displayed this time /7 https://t.co/l7oZiA2H05

If you've been following the threads in this series, you'll know that at some stage I'll talk about "the rules" on how you can and cannot order the various types of parameters in function definitions. Here's and example which breaks these rules: /8 https://t.co/FcIsaQkGKg

The error message, new in Python 3.11, makes it clear what's wrong in this case The variable keyword parameter, that's the **kwargs, must come after the "normal" parameters /9

We can even improve the function a bit by checking whether the name is already in the team and only adding it if it's not already there Notice how `"Stephen"` is already in the dictionary with a value of `4`, so it's not updated by the function /10 https://t.co/vvnKoUvU7j

Of course, whether you want to do this or not depends on the context of your program /11

Let's add another team and change the function so that you can add members to more than one team in a single function call /12 https://t.co/GPf1SrkQXy

You're now using both *args and **kwargs in this function When you call the function you can first use any number of arguments with no keyword followed by any number of keyword arguments /13

• The non-keyword arguments are assigned to the tuple `teams` • The keyword arguments are assigned to the dictionary `people` /14

Look at the outputs carefully and you'll note that Stephen, Kate, and Sharon are in both the red team and the blue team; Mary and Trevor are in the red and green teams; Ishaan is just in the blue team /15

Back to the rules of what you can and cannot do – note the function call which is different from the example above /16 https://t.co/oYW2TbO1yC

You cannot place the keyword arguments before the non-keyword (positional) arguments when you call the function This makes sense since `*teams` is before `**people` in the function definition /17

But can you swap the *args and **kwargs around in the function signature? Let's find out… /18 https://t.co/aXKb68ZCvy

No, you cannot Notice that the error is raised by the function definition and not the function call (there is no function call in this example!) *args must come before **kwargs in both the function signature and the function call /19

In summary, in this thread you've seen a few examples of functions with "normal" arguments, *args, and **kwargs There are more combinations of parameter types you could use /20

We now move on from *args and **kwargs Earlier in the series, you've seen how when you call a function with "normal" parameters (not *args or **kwargs), you have the choice of using positional arguments or keyword arguments when you call the function. /21

In the next two days we'll see how you can force the user who's calling a function to use only positional arguments and not keyword arguments, and the other way around, too /22
