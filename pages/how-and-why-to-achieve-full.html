How and why to achieve full-stack type safety in your web application, from the database through the API to the frontend. A thread on the beauty of #TypeScript ðŸ§µ (1/13)

TypeScript has been the most crucial tool to increase productivity within our agency. Combined with tools like IntelliSense, TypeScript allows us to skip many journeys to the documentation of OSS libraries and detect simple but harmful bugs before they're ever deployed. (2/13)

For a long time, type safety stopped at the borders between the backend, database, and REST API. To improvise on the latter, we often used private NPM packages containing shared type definitions for the API layer. However, maintaining them has always been a burden. (3/13)

That's why we've designed a new standard stack for our web and mobile apps using great open-source projects like Nx, Prisma, and OpenAPI. Here's our fresh set-up, engineered for maximum productivity and type safety. (4/13)

(1) Nx: Nx monorepos have been our #1 productivity accelerator, right after TypeScript. Remember the private NPM package which was used for shared API types? (5/13)

Deploying a new version of the package with every minor change to every app in order to ensure everything still adhered to the types had been a significant bottleneck. With Nx, we can create a shared lib within the repo without the overhead that comes with an NPM package. (6/13)

(2) OpenAPI: Writing and updating those types by hand after designing an API still didn't feel right, however. The great tool "openapi-typescript" allows us to design our API in a central OpenAPI spec and automatically generate type definitions from it after every change. (7/13)

Both the API server on the backend as well as the API client (again a shared library for every frontend app within the monorepo to use) directly make use of those generated interfaces. When there's a breaking change on either end, we immediately notice. (8/13)

(3) Prisma: Previously, most of our projects included some kind of hand-written SQL statements as model definitions. Not only has this felt like a hindrance in scalability, but also introduced a lot of bugs due to the entirely missing type safety and linting. (9/13)

We've always had an eye on Prisma, an open-source ORM, and felt like it had finally reached the point of being production-ready this year. With Prisma, we define our database models and enums directly in a schema file which then gets converted to TypeScript interfaces. (10/13)

We can then use a Prisma instance to communicate with the database. To sync the types between Prisma and the OpenAPI generator, we've written a small, handy tool that replaces some of the interfaces generated from the API spec with their Prisma counterpart. (11/13)

The outcome of all this? Much less boilerplate and maintenance work resulting in much more time available for actual productive work. (12/13)
