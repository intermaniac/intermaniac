I've spent quite some time building CLI tools in TypeScript and every library for prompting, I mean every, the provide you a strict API and almost no way to easily customise you rendering/output. In/Enquirer, prompts, ink, etc. /thread

First interaction was with Ink, that is an incredible library, built on top of Yoga layout engine, provides you a flexbox in terminal. I suppose it's a huge engineering effort and result is incredible, but I don't see how this is good for CLI..

CLI is usually a sequential communication between command line and developer. Declarative UI is an overkill for CLI. You have to write an obscure amount of code for simple stuff in terminal.

Ink has a massive amount of prebuilt components, but those are meant to be used as is, if you need to customise - you have to fork a particular one, learn how it works and change it. We've built an CLI app with ink and it was fun until we made a step back

When we wrote the same thing in an imperative manner codebase went like 70% down and business logic was way simpler and slimmer. At that point we started walking around Inquirer/Enquirer and prompts

They're really solid and being developed for many years and some of them adopted hooks, they're really nice but it's still hard to design it your way and that is something I can't wrap my hand around. I want a different look and feel https://t.co/Ug54lEJeEs

And every time I literally have to build my own. I bumped into oclif and they have a dedicated command line to generate a FOLDER for you to write your own component, sheesh

I've recently watched a video of @t3dotgg where he watched a video of @tannerlinsley giving a speech at react summit and talking about the future of @tan_stack. That idea of headless/designless did really solidified in my mind

When you have a designless/headless library - some developer has done 90% of heavy lifting for you and you now have freedom to lay it out the way you want. As opposed to headfull where you've got 100% done, but you're within strict boundaries now

I truly believe that in long run every time if you stick to head/design-less libraries you gonna benefit immensely. Wether it's backend or frontend. I've been doing and managing software development for quite a while and if you're not delusional

If you can make a step back and analyse - mostly every time you'll see that in a long run, using headless will benefit both you as an engineer and your product.

Okay, back to CLIs. Since I haven't found a sweet spot for my use case, I decided to build one. I just love building stuff, I can't stop doing that and I probably wont and I'm completely fine with it. So I've spent couple of hours on weekdays with internals of the cli libraries

Mainly everything comes down to obtaining stdin/stdout, observing keypresses and either halt or consume the input and render the output. When you render - you just clear what you've rendered before and render new stuff. It's so easy conceptually

The above you need when you need to prompt user. If you want to statically render the text - I encourage you to just console.log and use chalk or ansiicolors or kleur for styling, they're all awesome.

This is how you prepare readline and how you teardown when you're done with the prompt https://t.co/wZCcLZ3YqU

That's it, that's all you need from command line itself. Ah, maybe to just remove and show cursor if you're doing selects of something. There's a sisteransi library with all hex commands for command line operations https://t.co/YF0xldWimX

to render you just make a text https://t.co/n54IgeYKrw

So on library side I handle CL lifecycle and for prompts like Select or Input I will prepare a UI agnostic data bundle to handle all the "back", "forward", "up", "down", "delete", "return", etc And all you need is to just define how to render
