I love working with just stateless functions, interfaces and literal objects in #TypeScript becaus everything's so easy to understand and to test. But sometimes, when everything falls into place, classes can be beautiful, too! Let me explain... ðŸ§µ

I am currently re-writing the parser for @MarmotaApp so that it can partially parse a document based on a set of changes (when you edit the document and add a letter in the middle of a headline, it can parse only that headline, leaving the rest of the parsed document as-is).

This requires tracking a lot of state, and because of that, I decided to use classes for most of the code from the start. Then, I implemented partial parsing in two parsers as straight-forward as possible: One for "leaf" nodes and one for nodes that can contain other nodes.

I saw that the code was completely generic for both cases. And I wanted to extract it, so that no future parser implementation (e.g. for list items) never has to deal with partially parsing. But I wanted to make sure that my assumption ("it's completely generic") was valid...

So I implemented a dozen tests for partially parsing headings and checked that they fail with a reasonable error message (see https://t.co/n6xvGN740T ). But then, I immediately skipped all those tests and started the refactoring, with all tests green.

The remaining steps were quick and easy: * Create a base class for leaf parsers and use it -> tests green * Move code to parse the node itself to base class -> tests green * Create a base class for container parsers and use it -> tests green

* Move code to parse container node to base class -> all green * Remove duplicated code in container parser by calling leaf parser -> all green

And then, there was this moment where everything fell into place: I un-skipped the tests, made the heading parser extend the container parser, and all the previously-red tests were green!

Sometimes, class hierarchies (with interfaces and abstract base classes) can be beautiful. ("Prefer pure functions+data" and "Prefer composition over inheritance" still apply. "Prefer" just doesn't mean "Always use instead")
