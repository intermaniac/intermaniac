The default API Gateway error reporting for uncaught Lambda exceptions is terrible. You get a generic Internal Server Error message with 500 status code for everything. Use middleware like middy to wrap your Lambda handlers, catch errors and return a consistent error response.

Avoid throwing the built-in Error class. Instead, use something like VError to throw and subclass your errors, so that they are reported with a different name, have a formatted message, add extra context with info, and include the cause of the exception for error chaining.

When you combine VError with a custom Lambda middleware for exception handling, you unlock a powerful duo that you can really up your error reporting game. Some of the things you can do with VError:

1. Use VError.fullStack() for a full error chain In your custom middleware, use VError.fullStack() instead of just console.log to report your error. It will include any extra context you added (via the .info field), as well as the full error chain (via the .cause field).

2. Add descriptive fields to the VError object for API response Add these fields to the .info part of the VError you throw: statusCode - HTTP statusCode errorCode - one of an enumeration of error codes that are publicly documented in your API

3. Use those custom fields in your middleware to customise API response You can use these fields to customise the response HTTP code (statusCode) and an human readable error code in the body (errorCode), naturally leveraging JavaScript's exception mechanism

4. Hide internal errors Conversely, when those fields are absent, you know it was an internal error and you can just return a generic 500. Your logs will still have the full error chain from your middleware using VError.fullStack().

5. Add the exception to your current OpenTelemetry span If you're using OpenTelemetry in your AWS Lambda functions (and if you're not, you should be), you can leverage middleware again to add details from the uncaught exception to your current span.
