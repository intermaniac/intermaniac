The random oracle model (ROM) models idealized hash functions as public random functions and provides everyone with oracle access to them, opening up many new ways of arguing security that would be difficult or not known to be possible in the standard model. (2/)

In the ROM, it is impossible to use the code of the ideal hash function. There are a few, though not too many, techniques of using the code of a cryptographic algorithm, such as circuit garbling, obfuscation, functional encryption, etc. (3/)

Take garbling for example. What if we want to garble an idealized hash function? Consider a "usual" hash function. Due to the simulation security of the garbling, a garbling with one set of labels reveals nothing more than the value of the hash function at that point. (4/)

During simulation, indistinguishable from the real world, garbling a hash function is not really using its code, just a single call to it. Moreover, when a scheme in the ROM is instantiated, we replace public random functions by real-world hash functions, which do have code. (5/)

Intellectual curiosity mandates us to pose the question of how we can formally allow non-black-box usage of an idealized hash function. Our answer to this question is the *pseudorandom oracle model* (PrOM). (6/)

When cryptographers see "random functions", the neuron connected to "pseudorandom functions" are activated. The PrOM is just that --- it is defined relative to a (standard-model) pseudorandom function (PRF) family. It provides two interfaces --- (7/)

one to map a PRF key to a handle, the other to use a handle and a PRF input to evaluate the PRF on the corresponding key. The PRF key corresponds to the code of the hash function, and the handle corresponds to the oracle access. (8/)

We show that in the PrOM, *ideal obfuscation* (simulation-secure obfuscation, stronger than virtual black-box) can be constructed from a standard-model functional encryption, at a polynomial loss of security. (9/)

Its construction and security proof are an example of how to use the PrOM --- roughly speaking, the code of the hash function (read "PRF key") can be used in non-black-box constructions, and should be used in a protected manner (encrypted in our case), and (10/)

in the security proof, once a PRF key is information-theoretically removed (using functional encryption security in our case), the function corresponding to its handle can be switched to a truly random function, which "idealizes the hash function". (11/)
