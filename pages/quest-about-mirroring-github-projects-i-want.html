===Quest about mirroring GitHub projects=== I want to mirror my GitHub projects on other platforms for many reasons. This is a thread describing my difficulties with it and a working solution ðŸ§µ [1/n]

For a year, I used the GitHub->GitLab pull mirroring. It could copy the code, tags, and discussions in issues and pull requests, which was nice. But some time ago, I noticed that my GitLab branches fall behind the origin. I signed in to GitLab and saw this annoying banner [2/n] https://t.co/QgM8p0Q2Xu

GitLab pull mirroring disappeared from some of my repositories. Enabling it again with Free Tier is not possible anymore. Buying GitLab Premium Tier is not possible for anyone from Russia. Even a free trial is not allowed. So I started to look at other mirroring solutions [3/n] https://t.co/SbR6fLUqxg

I checked Chinese Gitee and didn't like its limited support for English localization: https://t.co/tihayowYHF I checked the Radicle p2p network for software development. But its "powerful blockchain-based functionalities" looked too radical to me: https://t.co/IYsKlhxJKl [4/n]

Then I looked at https://t.co/qv8b9udsyI. It's driven by a non-profit organization promoting FOSS ideas. I like them much more than Microsoft. But in March 2020, they disabled mirroring due to a lack of resources: https://t.co/XPD5ctMy1d :( [5/n]

I created the mirrors for my GitHub projects at GitFlic, a small proof-of-concept code collaboration platform. https://t.co/Qbyp3BaC1Q But it doesn't have CI and can't copy the information from GitHub issues and pull requests. Obviously, not a final solution. [6/n] https://t.co/b5YmlMO27D

To sum up, I didn't find any code collaboration platform that could provide full-featured mirrors for my GitHub projects. So I decided to look at this task from another angle: how can I manually back up the info from the GitHub issues and pull requests? [7/n]

My first idea was to turn the discussions into a part of the code. Shortly, I found an excellent project gh2md, which can help with it: https://t.co/QnoQV38rKo gh2md grabs the information from GitHub issues and pull requests and turns it into a Markdown document. [8/n]

Under the hood, it uses GraphQL API provided by GitHub, so I had to generate a GitHub personal access token with public access. More info: https://t.co/BLcJKwYXlp Now my projects contain the https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3 file with a backup of issues/PRs: https://t.co/94dDHcRugh [9/n] https://t.co/phj5WChKJB

For a moment, I thought about a GitHub Actions script generating a pull request with an https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3 update. Shortly, I realized that it was not a clever idea: the PR would make this CI script update https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3">https://t.co/Fsavlbvtn3 once again and generate another PR :) [10/n]

Anyway, after committing the Markdown backup of GitHub issues and PRs, I decided to create clone repositories at Codeberg and then do a git push both to GitHub and Codeberg. GitHub->Codeberg migration of my projects worked well. It copied all the info from the origin. [11/n]

So Codeberg would be the ideal solution for me if it had a pull mirroring feature... Alas! But what to do with out-of-date issues/PRs at Codeberg? Manually deleting and recreating projects would be a dirty hack, I didn't like it. [12/n] https://t.co/jvNORrIFWc

I looked through repo settings and found a workaround: Codeberg provides options for using an external issue tracker! I set the URL format and numbering for the external issues, disabled the Codeberg pull requests, and now have the correct links to GitHub issues and PRs. [13/n] https://t.co/0DYEGN3Ig4

So if something goes wrong with GitHub, I will enable the internal issue tracker and pull requests at Codeberg. That's it ðŸ§µ Maybe this story will be useful to somebody. Any ideas or comments? [14/14]
