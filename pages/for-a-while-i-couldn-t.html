For a while I couldn't publish new slide decks on https://t.co/54Z1QInoEh because... well, all these were involved: * stupid duplications * a 4.5GB TAR file * GitHub Action timeouts Here's what happened. 🧵 (overly long)

Background: Each of my slide decks is a little website that is generated from Asciidoc files and displayed/animated with reveal.js and highlight.js. Everything is text-based and very amenable to version control, scripting, etc. I can only recommend that!

Btw, here's the repo: https://t.co/ujkKzka6Go Take everything! The presentations themselves (i.e. .adoc and .html files) are under CC BY-NC 4.0 but this tweet licenses my source code (.js files in root folder) under WTFPL. Go crazy!

Back to the topic: Slides, images, theme, even reveal.js and highlight.js evolve over time but for every presentation I give, I want to have a permalink that conserves the slides exactly as they were. What's the simplest way to do that? Just create a self-contained copy!

So that would be the presentation itself (an HTML file), the images it contains as well as reveal.js and highlight.js. Apparently that averages 33 MB per copy. Multiply by 155 talks/presentations and I end up with 5.1 GB. 😳 That turned out to be an issue. https://t.co/DEsSTdcjdJ

Because GitHub Pages' deployment didn't like that. It goes like this: 1. checks out the repo (the 5.1 GB) 2. creates a TAR (~4.5 GB) 3. sends it over the network 4. unpacks it on the web server 1.-3. took about 40 min. 4. times out at 10 min. 😭 https://t.co/8w1cdYuP9q https://t.co/prT0BiQqLh

So there you go, no more publication of changes to these talks or of past presentations. For two years, I was waiting for this to happen and of course it did at a time when I was under water and couldn't fix it. 🤬 Still, it motivated me to tackle it ASAP, which was this week.

The obvious issue is that images and reveal.js/highlight.js versions rarely change between presentations and are thus identical copies in many folders. Solution: Just put them in a shared place and reference them there. https://t.co/FBhcVezRAY

Side note: This works so well because while Git stores the entire repo history, it only stores each file once. So the duplication is not present in .git, just in the checked-out folder. Fix it there and it's all good.

But images can change while keeping their name or have the same name in different talks in the first place. And likewise reveal.js/highlight.js evolve. So it's not quite as simple as copying to a shared folder. The (obvious) solution is to append a file/folder hash to each name.

Hashing a file is simple enough in JS: const f = require(`fs-extra`).readFileSync(file) const hash = require('crypto').createHash('sha256') hash.update(f) return hash.digest('hex') That takes care of the images, but what about hashing the reveal.js/highlight.js folders?

Some NPM packages can do that but I rely on Git anyway, so why not use it? A quick search shows how to hash a folder: git ls-files -s | git hash-object --stdin Neat! I ❤️ Git!

The results is a folder structure like this: _slides ├─ _highlight.js │ ├─ 376a7e15 │ ├─ 8fbb4352 │ └─ head ├─ _images ├─ _reveal.js │ ├─ 05e6ee1a │ ├─ c7ffc242 │ └─ head ├─ _images ├─ # talk folders └─ # other files

Now some search and replace in the generated HTML file to update paths to images and folders! I ended up calling `replaceAll` once for each changed path, which is mildly annoying because it appears wasteful. But it doesn't really matter in practice, so I won't optimize it.

And the result? * site size down from 5.1 GB to 380 MB * build times down from 40 minutes to 4 https://t.co/3AjXlbhMKK * giving an unchanged talk only adds a single HTML file 🙌🏾 Oh, and a commit that's too much for GitHub to handle. https://t.co/V1vAumUbF5 😁 🏁
