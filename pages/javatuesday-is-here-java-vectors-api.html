The problem is that only one element of the data set is processed. This approach is acceptable as long as the overhead of the computing infrastructure does not exceed the cost of the computation itself (excluding memory). https://t.co/z8PsfERmMP

In an attempt to create an efficient array allocation in Streams, more time is spent on segmentation than on computation, and the structures themselves become more complex. But there is a way out - to use the SIMD procedures.

Typically, SIMDs include basic arithmetic operations such as addition, subtraction, multiplication, negation, absolute root (abs), square root (sqrt) and the already mentioned FMA, and many others. From the development point of view, vector operations look the same as C/C++ code: https://t.co/Pa3iRO5XAf

In contrast to C functions, vector intrinsics are implemented directly in compilers. The above SSE insertion is usually compiled into a single instruction.

Modern compilers and libraries already implement many things using SIMD, assembly language, or a combination of both. For example, some implementations of the standard C library routines memset, memcpy or memmove use SSE2 instructions for performance reasons.

However, outside of such niche areas as high-performance computing, game or compiler development, multimedia, even very experienced C and C++ programmers are mostly unfamiliar with SIMD features.

Before the implementation of OpenJDK Project Panama, there were no vectors in Java at all, in any form, except for access through JNI.

Although access to processor vectorization is possible directly through the Foreign Function & Memory API or JNI, it is not the only one. For quite a long time, Project Panama has been developing native support for vectorization.

The whole issue of vectorization and vector processing revolves around the issue of performance. It was mentioned above in the article that the speed of the same vector within the usual for loop and the corresponding processing with SIMD is linear. Is it really so?

Let's do the following synthetic test: take two arrays of type double of dimension 1M elements and implement the addition procedure using "vector sum": https://t.co/qTILRYoXRH

... "for-loop sum": https://t.co/4GfmJ5EUDD

... "stream-based sum": https://t.co/05oFLI4XDd

JMH gives the following results for these implementations on Apple M1 (128-bit vector operations): https://t.co/NkpIIocW1O

These results still show that explicit and auto-vectorization have almost the same execution time, at the same time, we see a slightly different relative execution time of the addition in the streams, and in fact, the ratio is slightly different - 8.4x times!

Generally, once your code becomes "hot" the execution time will significantly improve over time. In my case, from 200ms to 0.4-0.47ms. https://t.co/CkGnDSmiF2

From the programming point of view, we see that such a result indicates the fact that the infrastructure overhead is very high, and it also means that there is an extremely high need to test the running time of each algorithm that can potentially be optimized by the C2 compiler.
