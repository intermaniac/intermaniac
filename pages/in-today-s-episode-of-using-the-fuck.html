As you can see from the snippet above, our __iter__ dunder generates the next integer until it reaches our stop specified, which was not the case in Python 2. if I set the stop arg to be 1 Googol (10^100), there won't be enough memory to store the array of integers.

And as you know, this would have caused a memory problem. but the real magic doesn't stop there. upon further fuckery, we can see from the output of the for loop, the first value should compute 0^0 [2...]

but wait, 0^0 does not make sense. we know for b^0 for any value b, where b != 0 should be 1. but in this case, the value for 0^0 was 1. This of course we revealed our "Find Out" for the day [3..]

The internet told me per the IEEE 754-2008 floating-point standard, programming languages have their own way of dealing with this atrocity to mathematics. The C and C++ standards do not specify the result of 0^0 (a domain error may occur), of course, it had to be C & C++.

but The Java standard, the .NET Framework method System.Math.Pow, Julia, and Python also treat 0^0 as 1, which breaks math for me but it's alright. I wonder how the math community compensates for this condition. But to me, this was a shocking surprise when I run that code.
