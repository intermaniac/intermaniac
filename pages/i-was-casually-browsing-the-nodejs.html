I was casually browsing the @nodejs https://t.co/QyUYmOW6mm website when I noticed this behavior: A thread üßµ https://t.co/VaaSL9bcQZ

On further investigation, I noticed the script responsible for building the 'Edit on GitHub' link was adding https://t.co/tWI0Pl8rcJ to all pages even though some were pages like https://t.co/3H112WrD7s">https://t.co/3H112WrD7s https://t.co/o8wVAOn754

I believe the culprit is in this githubLinks function. But right now I don't know what to do yet https://t.co/b42xZpqITI

On further investigation, I noticed all path had a /index.html appended to them so that means when the paths are constructed for the GitHub links, https://t.co/3H112WrD7s">https://t.co/3H112WrD7s becomes code-and-learn/index.md I decided to split the path and take a look. https://t.co/BlZG2LQVSX

The pattern I noticed was for pages(after I split the path into an array) with length <= 2, the path will work. It won't however work for >= 3 so I decided to pop the index.html for such pages https://t.co/Wv7eb6LU5P

The above worked but what happens when a nested page does have https://t.co/tWI0Pl8rcJ? Yeah you get it the algorithm above will throw that away so that ain't good. Back to the drawing board, it is... https://t.co/EDn8E5miD4

So after a couple of minutes of thinking the hard way to fix this, I had an epiphany: since the githubLinks function is a middleware, what if I can call it before whatever turns the .md files into .html files is called. https://t.co/PNfITPxN9R

So I moved the .use calling githubLinks to another spot in the chain of .use and it worked, the files where no longer being transformed and I don't need any manipulations in the githubLinks middleware to get the correct file for GitHub edits https://t.co/Y37H2Fw1xV

So the above solution was nearly perfect but the "Edit in GitHub" button disappeared and I should have predicted this because if you call the githubLinks middleware before layouts, we won't be able to inject the button since the footer is in the layout ü§¶üèæ‚Äç‚ôÇÔ∏è I'm going to bed...

Okay fine, I couldn't sleep because of this fix so I ended up researching on my phone about metalsmith - which is the static site generator being used here. It turns out it has a bunch of plugin and the one responsible for turning about.html into about/index.html is...

the permalink plugin https://t.co/JG7qtSsi7h

All I needed to do was to move the middleware calling that plugin after the one for both layouts and githubLinks and that solved it. https://t.co/KDVV7al4zK

Here is the navigation after the fix. https://t.co/X4YkTOHELI

Submitting a PR to suggest this change. https://t.co/etkciy54rL

Lessons: it was fun trying to figure out what was going on even though I have never used metalsmith before. I might have saved time by studying the metalsmith plugins but going down the rabbit hold and experimenting until I found a fix was exciting.

So I ended up working some more on the PR because a few pages were broken. I went with an earlier implementation that was removed a while a go(client-side JavaScript was needed to make it work) But let's see.

I'm quite interested in seeing how this can work without the client-side JavaScript and not traversing the locales/ folder without metalsmith. Interesting stuff.
