2/10 - Use case Always make sure a #main and #dev branch stays in sync. The workflow would be PRs from a feature branch into dev, and then a PR from dev into main.

3/10 - Initially, I merged PRs from feature branch to dev with a normal #merge commit, and then used the same flow from dev into main. The issue was that dev kept falling further and further behind main, as dev would never contain the merge commits that ended up in main.

4/10 - Side note: Although having nothing to to do with the actual use case, instead of creating a merge commit when merging into dev, I now use "Squash and merge" to neatly package the whole PR into a single, new commit.

5/10 - To solve the issue of merging dev into main, I thought using "Rebase and merge" would do the trick. If main is a couple of commits behind dev, just take those and apply them on top of main, and we're good, right?

6/10 - Not quite. Apparently, Githubs rebase differ slightly from git #rebase. "Rebase and merge on GitHub will always update the committer information and create new commit SHAs" https://t.co/CJVKdy9el9

7/10 - Again, dev and main kept growing apart in their history. Eventually, I turned to @github Actions, with the thought of using plain git to rebase dev from main anytime something happened in main.

8/10 - The current setup now looks like this: feature branch -> dev: "Squash and merge" dev -> main: "Merge pull request" (with merge commit) Github Action: rebase dev from main (to bring the newly merge commit over)
