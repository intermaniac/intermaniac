In ECS, you completely separate behavior from state. ‚ñ´Ô∏è ENTITIES are "things" in your world (player, enemy, coins, etc...) ‚ñ´Ô∏è COMPONENTS are characteristics of your entity (position, velocity, etc...) ‚ñ´Ô∏è SYSTEMS are the behaviors (logic) that you apply to components /2

Components are linked to entities and can be reused. Example: ‚ñ´Ô∏è 3 entities: Player, Enemy1, SolidBlock ‚ñ´Ô∏è Add a POSITION component w/ X,Y,Z to all entities ‚ñ´Ô∏è Add a VELOCITY component w/ X,Y,Z only to the Player and Enemy /3

Systems iterate over a set of combined components. ‚ñ´Ô∏è A "Movement System" will iterate over all entities that have both a POSITION and VELOCITY components ‚ñ´Ô∏èThe SolidBock entity is ignored by because it doesn't have the VELOCITY component /4

This gives us a lot more flexibility to compose systems and reuse both the COMPONENTS and the SYSTEMS in unique ways. For example... /5

I wanted to debug my bounding boxes for player and solid entities. Simple enough. ‚ñ´Ô∏èAdd a DebugCollision component to my Player entity with one property (color). ‚ñ´Ô∏è Add a System that iterates over any DebugCollision components and renders a graphic over the entity. /6 https://t.co/iJAawVFqYi

When I wanted to see the bounding boxes for all my solids in the scene, I added the DebugCollision component to the solid entities. The system automatically generated the overlays. /7 https://t.co/2Kxjnlig2H

In summary, ECS architecture isn't for all projects, but once you wrap your head around the concepts it can be a powerful way to create modular code that separates concerns. Link to the GDC Vault for the presentation: https://t.co/I5IDqnwu2a 8/8
