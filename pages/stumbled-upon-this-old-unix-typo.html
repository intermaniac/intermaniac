Stumbled upon this old Unix 'typo' code, written in the 1970s, on @rob_pike's github: https://t.co/MpBMNb3drv (also at https://t.co/FmOwsNf3pt, man page at https://t.co/EHqbkpwCtX). This is the program where the first streaming algorithm, the "Morris Counter", was invented. ðŸ§µ 1/

We covered randomized algorithms in my class last+this week, so I projected a portion of the code today to show students a randomized algorithm in action. Morris wrote a language-independent spellchecker, which counts bigrams and trigrams in words (see lines 100-105). 2/

His trigram array 'tab3' has size 19,684 =~ 27^3, and apparently his machine at the time could only afford 8 bits per array entry for such an array size. But some #'s needed > 8 bits! Thus when incrementing count x, he did so w/ probability ~ .968^x (line 40). Why .968? Well.. 3/

Incrementing with prob. 1.0^x is just a regular counter (no variance, high memory). 0.5^x would give low mem (loglog N bits for value N), but high variance. Natural question: what about (1-a)^x for a small. Can we smoothly trade off mem for variance? This is what typo did! 4/

Morris wasn't creating clever randomized algorithms just for the heck of it; he was doing it out of necessity. An example of great engineering! And what's more, his idea still lives on. Yes computers these days can afford log N bits to store a number N for any reasonable N. 5/

But even in the 70s, Morris didn't need to store just 1 counter, but ~20k ctrs. Then (bits per counter) is a multiplicative overhead on (# counters), which is big, so best to keep it small. That use case still exists today in Redis, a popular in-mem DB often used as a cache. 6/

A cache needs a cache replacement policy, and one implemented in Redis is "Least Frequently Used", which needs a counter for each key. So if the DB has many keys, that's many counters. These counters are maintained using a variant of the Morris Counter: https://t.co/jvHJlclSYt 7/

This algorithm is close to my heart; Huacheng Yu noticed a gap in the literature for upper vs. lower bounds ~2 yrs ago. He told me about it, and we worked on it. The conventional analysis was that if you want 1+t approximation with confidence 1-p, Morris' solution would need 8/

O(loglog N + log(1/p) + log(1/t)) bits for counter value N. We showed throwing a heuristic on top of Morris (which happens to already be implemented in Redis!) can improve the log(1/p) to loglog(1/p), and furthermore this is optimal for any algorithm. The heuristic is natural: 9/

use a deterministic counter up to a certain point, then switch to being probabilistic (realized in Redis by only doing probabilistic increment if counter >= LFU_INIT_VAL). But this doesn't quite address the situations of both Morris and Redis (as pointed out by Greg Valiant). 10/

Morris/Redis want to not just maintain one counter, but M of them (e.g. M=27^3). Maybe there's a more clever amortized solution than storing M independent Morris counters?!?! With @AdenIshaq, @yanjun_han, and Huacheng, we show no such solution exists https://t.co/Wmotxq2WtF. 11/

There's still a small gap; our impossibility thm holds for *almost* all settings of parameters, but needing the restriction t < 1/lglg(1/p). So, there's still a narrow range of t (basically, the regime of constant approximation) still open. Maybe one of you will resolve it! 12/
