At last I fixed/updated my Github action deploying my static blog to Github Pages. A small thread üßµto read if you wonder what is the pages-build-deployment (and are possibly annoyed by it) https://t.co/KpU6Gdyceh

Originally (or at least when I started to use it), deploying a static website on Github pages consisted in pushing the HTML/CSS/JS on the gh-pages branch.

At first, I used Travis CI to build my Jekyll website and push it to the gh-pages branch because it was using some custom plugins, which was not supported in standard GH pages until Github Actions arrived.

Once Github Actions was there, I was able to build my Jekyll website with its custom plugins using docker. I moved from Travis because I wanted to add some Cloudflare cache reset and wanted to test GH actions https://t.co/QRm20lObQT

Lately (well, a few months ago), I noticed a pages-build-deployment actions that was running once my own action finished. It was annoying because it was rebuilding my site and my cache cleaning was ineffective because done BEFORE actual deploy. https://t.co/tYjDCr2lCl

I had no clue how to fixed that (and had other things to do than search for it üòÖ) but by chance, a few days ago I noticed a modification in the pages settings of my blog's Github repository that helped me solve my problem.

In the Pages settings of repository, you can choose between "Deploy from a branch" (that launches the pages-build-deployment action when there's a push on gh-pages branch) or (custom) Github Actions https://t.co/ysPHL1Sbgx

All I needed to do was to check "Github Actions" as deployment source and do in my custom action what is done the the pages-build-deployment action

As after enabling GH actions it shows a new one, I thought I had to set a specific Github action, but no. This mode just need that an action triggered when you want: when a push is done on may, on cron (schedule), or manually (workflow_dispatch) https://t.co/kC6dxMdGHE
