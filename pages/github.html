GitHub is a front end for git, a way of collaborating on files. It's similar to DropBox or SharePoint, but git is VERY focused on versioning and merging different versions of files. DropBox stores your files; git tracks your changes.

At its heart, git is a tree. Every "repository" has a tree. The trunk of the tree is called "main". It's usually stable source for whatever project is being developed. "Branches" can be stable, unstable, or something else, like documentation. "Main" is a branch, too.

Like a tree, you generally start with the trunk ("main") before exploring branches. Climbing out on a narrow limb (an experimental branch) can get you in trouble. Those branches are very useful, however.

Right now, for example, OmniBox has - The main branch - The 0.9.9 branch - An 0.9.9-experimental branch - & A branch for documentation web pages I created the 0.9.9 branch to submit changes for that release. This keeps them out of main until they are ready.

Similarly, I wanted to keep the changes involving adding heat set inserts out of 0.9.9, so I created the -experimental branch to track my changes there. 0.9.9-experimental is a branch of 0.9.9, much as a tree can branch multiple times.

UNLIKE a tree, you can take changes from branch A and merge them into Branch B. Branch B can be the trunk or another branch entirely. Git does not care, only that it understands how you want the branches merged. This gets complicated, but it's important.

Let's say that I "commit" (submit) 20 new files to my 0.9.9-experimental branch, but only 18 work out. - I can roll back my change to the two 2 bad files (remove them). - Or I can merge only the 18 good ones into the 0.9.9 branch (leave the 2 behind & not use them).

Hopefully that's clear. The idea is that I'm filtering the good stuff from the experimental branch to 0.9.9. When 0.9.9 is done, I'll merge it into main. When you download the latest release package, it's made with the files in main. /end
