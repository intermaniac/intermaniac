Webpack 101 @webpack is a static module bundler for modern JavaScript applications that scans through your code, internally builds a dependency graph from one or more entry points, and combines every module your project needs into one or more bundles. #javascript A thread

These bundles are static assets that serve your content. A dependency graph is (literally) a graph that includes every module an application needs. Webpack uses this to bundle all those modules into a small number of bundles often only one - to be loaded by the browser. 1/15

To generate a production-ready build using webpack cli, we can use the given command. All it does is to takes our index.js as the entry point for our project and convert its content to something ready for the production environment. 2/15 https://t.co/nOb0WhZf7D

Output: dist (aka distribution) directory will be created at the root of the project that will have a main.js file (by default) that will be a transpiled (browser compatible) and minified version of our code. 3/15

Running commands is all fine however when we want to specify and do a lot more, config files become a necessity, especially when working with others. It's way easier to debug a config file than an exceptionally verbose command. In order to compile, we can simply run this. 4/15 https://t.co/LeQUGN9R82

Let's go over the core concepts individually. 1. Entry An entry indicates which module, webpack should use to begin building out its internal dependency graph. Webpack figures out which other modules and libraries that entry point depends on (directly and indirectly). 5/15

By default, its value is ./src/index.js, but you can specify different (or multiple) entry points by setting an entry property in the webpack configuration. All this does is inform webpack from where our app will start. 6/15 https://t.co/nMvPsaUr2E

2. Output This property tells webpack where to emit (save) the bundles it creates and how to name these files. It defaults to `./dist/main.js` for the main output file and to the `./dist` folder for any other generated file. 7/15

You can configure this part of the process by specifying an `output` field in your configuration: In the example above, we use the output.filename and the output.path properties to tell webpack the name of our bundle and where we want it to be emitted to. 8/15 https://t.co/uVy5oRh0BP

3. Loaders Webpack by default only understands JavaScript and JSON files. **Loaders** allow webpack to process other types of files and convert them into valid modules that can be consumed by your application and added to the dependency graph. 9/15

At a high level, loaders have two properties in your webpack configuration: - test: identifies which file or files should be transformed. - use: indicates which loader should be used to do the transforming. 10/15

This tells webpack's compiler to look for files that have .txt extension and use raw-loader to transform them into valid modules and then add them to the dependency graph. 11/15 https://t.co/6LJvA4hAH9

4. Plugins Plugins can add a wide array of functionality like bundle optimization, asset management, etc. Up until now, we were supposed to be updating the script tag in the index.html to point to the newly generated bundle in the dist, this is where a plugin can help. 12/15

With the help of html-webpack-plugin, we can tell webpack to generate a new index.html file automatically and inject the generated bundle after any change. In order to use a plugin, you need to install it, require() it, and then add it to the plugins array. 13/15

The html-webpack-plugin generates an HTML file for your application and automatically injects all your generated bundles into this file. 14/15 https://t.co/f7PEAFFfym

5. Mode By setting the mode parameter to either development, production, or none, you can enable webpack's built-in optimizations that correspond to each environment. The default value is `production`. 15/15 https://t.co/ihhi2daNQK
