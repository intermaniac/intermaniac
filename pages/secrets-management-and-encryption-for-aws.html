Secrets Management and Encryption for @AWS, @googlecloud and @Azure compared. A üßµ 1/

First, let's start with @AWS, and with encryption - KMS is a service that cuts across your entire AWS account. - Server-side encryption for S3, SNS, SQS, Dynamo, etc are managed by AWS Managed Keys - Generate symmetric, asymmetric and asymmetric sig only keys 2/

AWS has two services for Secrets Management: - Secrets Manager => Slightly more expensive. Allows key rotation. Cross Account access üëç - SSM Parameter Store => Cheap. No rotation. No Cross-Account Access 3/

As with everything else in AWS, you have IAM policies with actions that you can use to restrict access. You also have more localized Key Grants for KMS keys. You can use Resource Based Policies for Secrets Manager 4/

@Azure has an integrated service called KeyVault. For Secrets, Keys and Certs. - You can only generate asymmetric keys to encrypt and/or sign data - No symmetric keys unless you use their HSM service - Access control with Vault Access Policies or Azure RBAC 5/

@googlecloud's service is close to AWS Secrets Manager and KMS APIs. GCP ties all services with with its central IAM, which IMO, is easier to work with than having to manage several resource policies. Also I like the fact that you can use a Customer Managed Key for crypto ops
