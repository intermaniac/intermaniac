So yesterday I worked with AWS Lambda to implement one of the product. Well its just the beginning and would be interesting to see where this leads. However, I got to learn a ton of stuff. ðŸ§µ

What made it really interesting is that I chose a Go runtime on Lambda. I did not know that that Lambdaâ€™s code editor does not support Go. So the compiled go binary has to be zipped and uploaded to the Lambda to make it work.

One more reason for me not knowing the above fact is that I am using Terraform in parallel to do all this stuff. Terraform enabled me to deply the code as it is (without binary). Of course it fid not work.

Terraform zips the code files and uploads it to the Lambda, irrespective of the runtime. But thats where I found my naivety. I totally skipped the compile process specific to Go. After going thorough the documentation I found the right way to do it.

Then I ran into another classic challenge of handler name mismatch. That surprisingly took me a while to figure out. But it worked.

I am happy with the outcome. My repo is looking unusual, as the Terraform code to manage the Lambda function is bundled with the function. It somehow gives me the ability to deploy the code automatically to Lambda without me having to log into the console.

Also I made sure the Terraform code is not part of the compiled binary. Of course.

The way I did it for this Go serverless function, has given me a template to do the same for more such functions. That is a good feeling.

Next I need to think about managing the Terraform code for this Lambda function. Also need to think about Terraform code used to manage other components like API Gateway, VPCs etc.
