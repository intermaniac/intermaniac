Vector databases efficiently retrieve embedding vectors based on similarity, which is useful for neural search applications. However, vector databases represent data points as vectors, which may not be ideal for real-world data! Well, because as you know üëâ Real world is messy

Real-world information is generally available in nested form üï∏ For example - Let's consider a database of listings of an online store. Each listing, in turn, contains multiple images and a product description. How can we make this data searchable? üëâ DocArray to the rescue!

DocArray comes with these three features üëâ üìç Multimodal documents to model our data. üìç Subindices to make our data points and parts of those data points searchable. üìç Document Stores to store our data on disk (and efficiently retrieve it from there)

Model to use üëá CLIP(Contrastive Language-Image Pre-Training) by @OpenAI. You can simply use CLIP-as-Service to create embeddings with a simple API call.

Let's understand what the data should look like to solve this problem üëá Each listing needs an embedding: - To represents its description. - For every image in that listing. - To represent the listing in its entirety. https://t.co/Sh0PA2owc0

A DocumentArray is a data structure that stores data in the form of Documents. It can be used to persist data on disk. Each DocumentArray stores information for one search index, so a query can be used to find matching documents in it which can be loaded into the memory.

Subindices can be used to extend the search capabilities of a DocumentArray. Each subindex represents one nested level of the parent DocumentArray, such as an image or description. This allows you to search without loading all the data into memory first.
