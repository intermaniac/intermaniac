Keeping your data assets secure requires a multi-layered approach. Hardening your SQL Servers is a key part of that approach. So, what should you examine to help keep your SQL Servers Secure? A thread 1/12

In April 2020 The Motley Fool reported that Garmin paid $10M for a decryption key. When Baltimore was hit with ransomware, it cost the city an estimated 18.2M. Breaches are a lucrative business for hackers. 2/12

To help protect your SQL Servers and secure your data, we recommend reviewing the following: Physical Security. Limit physical and/or logical access to your SQL Servers. 3/12

Operating System Security. Ensure that the Windows or Linux operating systems are secure and patched. Patch. Cumulative Updates and other patches should be applied not only for performance reasons, but for security updates as well. 4/12

Services Accounts. Verify that the SQL Server service accounts have adequate restrictions, are not of Domain Admins, and change regularly. Consider using Group Managed Service Accounts when possible. 5/12

SQL Server Configuration Options. Limit configuration options such as CLR, Ole Automation, and xp_cmdshell on your systems. Don’t install any unneeded components such as SQL Server Reporting Services, or SQL Server Integration Services. 6/12

Encryption. Consider encryption; data lost due to bypassed security controls may be useless if encrypted. On a regular basis, audit the following: Roles and Permissions. Assess server level and database level role membership. Limit membership in the sysadmin role. 7/12

Authentication. Use Windows Authentication where possible and enable Password Policies for SQL Server Logins. Enable failed login auditing. Strong sa Password. Ensure that the password used for the sa login is strong. Change it frequently. Consider disabling the login. 8/12

Backup Strategy. Review your backup strategies and regular perform test restores to lower systems, including running integrity checks. Secure your backups. 9/12

To further protect your SQL Servers, consider implementing the built-in SQL Audit to record changes to important security facets such as additions to security role membership, new logins are created, and impersonation operations. 10/12

If you have questions about SQL Audit, we have some demo scripts available to help you to get started. Feel free to contact us for the scripts. 11/12

By reviewing and auditing your SQL Servers, you’ll have a greater sense of confidence that they are protected and secure, and you’ll help to take the guesswork out of managing them. 12/12
