At Ergo, you set up CI in seconds because we generate configs using your own build scripts. https://t.co/SlYjYtQbaD

ü¶Ä 2. Users should have fast feedback and not have to do a thousand git pushes The biggest problem with CI is how long it takes for engineers to receive feedback. With Ergomake, we allow you to run your builds on containers in your machine or trigger remote builds on the cloud. https://t.co/A3wKbdRxww

Think of Ergomake as "make on steroids". We make it easy for you to write reproducible workflows, and it just happens that they can run locally or on the cloud and integrate with your GitHub PRs.

ü¶Ä 3. Debugging workflows should be as easy as debugging local commands Every workflow is containerized, so we can put you straight into a container when something goes wrong. There, the shell is yours to debug. Once you're done, we give you the files back, ready to commit. https://t.co/n1GMX1BV6L

ü¶Ä 4. Setting up services for integration and end-to-end tests should be easy both locally and on the cloud With ErgoModules, you can set up databases and other pieces of software for your tests to use. We take care of spinning their containers up and making them accessible.

ErgoModules also come straight from GitHub, meaning you can publish yours and reuse others'. Ergomake is your default integration test tool, and it just happens that you can reuse the same configs on CI.

ü¶Ä 5. Generate TOML, not YAML At Ergomake, we want you to manage all your workflows through the CLI. However, we do see value in committing your CI configs to your repo. That's why Ergomake generates TOML, not YAML. https://t.co/h42YZr81yV

We do that because indentation is not a great way to convey semantics (we're sorry, Pythonists). Although YAML is a clean configuration format, TOML is simply cleaner, more readable, and doesn't depend on indentation. Also, we're Rustaceans, aren't we?
