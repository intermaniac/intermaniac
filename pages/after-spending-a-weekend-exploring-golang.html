After spending a weekend exploring @golang, I am tempted to use it at work, at least for some problems. I compared it with #Typescript and #Ruby, the other two languages in our backend. I am new to TS. Performance was not my primary reason to explore Go. ðŸ§µ ðŸ‘‡ 1/13

The decision to use a new language isn't what it used to be years ago. Tools like CDK, with services from @awscloud, enable us to use multiple languages seamlessly. Nevertheless, each new language adds overhead in toolchains and plumbing (structs, test factories, etc). 2/13

I am interested in learning a typed language that is quick to master. Typescript is an extensive language with a massive ecosystem, which can be great with experience, but increases the time to mastery. Go's syntax is small, so it's possible to master it in a few weeks. 3/13

I also won't be working in isolation. A strong community shapes the idiomatic usage, the package ecosystem, and the quality of engineers. It also influences your mental models. Gophers seem like a cool set of people. Check out https://t.co/G52SuT89lP 4/13

I want the @magicbell_io engineering culture to be more than a shared set of values. Conventions and tooling should enable better code reviews, robust deployments, and instrumentation. Again, less is more! Ex: Unlike JS, the best way to do imports is not a discussion. 5/13

This brings us to package management. I understand the package management hatred in other languages after trying Go. The Go code could be in CDK or another monorepo, but it compiles quickly and contains no unnecessary packages. It makes testing and deployments super easy! 6/13

Fast compilation leads to blazingly fast test runs. For a complex project, the speed of running a test suite becomes a blocker, no matter how much you try to parallelize it. It also becomes expensive with a growing team committing all day! The jury is still out on thisðŸ¤ž 7/13

Software is written to be rewritten, and anything that slows down change is unwelcome. Go lacks classes or other OOP primitives. While OOP is powerful, it's often misused to build opaque class hierarchies to get the job done. Go offers fewer ways to shoot yourself! 8/13

Widely accepted conventions in the community also allow for more readable and understandable code. This remains a major plus for #Ruby to this day, though it often leads engineers to stick to decade-old patterns. I'll have to see how this plays out if we end up using Go. 9/13

Ultimately, language is a tool for thought. I believe in modeling problem domains well, which requires an upfront investment, and rigorous standards for maintaining the model. This snippet from the Go book resonated strongly with me. So, why Go? 10/13 https://t.co/uGL1W0bZwt

The lesser time spent mastering the language and debating the best way to do everyday tasks, the more time you can spend on improving your domain model. This might be the only argument for the verbosity of Go (which is painful). it's a deterrent to quick hacks. 11/13

I don't have a lot to say about performance (yet). I can see how channels and subroutines lend themselves to higher performance in our multi-core cloud-native world. First-class support for concurrency should also avoid nasty bugs that bit us once in a while with Ruby. 12/13

Of course, it's not all Go! Check out this thread for arguments against Go from programmers I respect. https://t.co/UqFwymRlSH Many languages offer better performance and better type systems and I am sure Go won't be the last language we add to our Stack! 13/13
