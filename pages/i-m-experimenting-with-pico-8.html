I'm experimenting with PICO-8 and Lua. (having a lot of fun btw). I decided to make something "simple" like Conways Game of Life and boy was I proven wrong at how complicated this can be. ðŸ§µ Demo: https://t.co/UsqV7087Ac Github: https://t.co/wXkp3dybN3 https://t.co/GgD7tqLrXr

For those not familiar with Conways Game of Life, it's a life simulator that follows 3 simple rules to turn pixels on / off. https://t.co/laUMIBmGoU

First problem was calculating the neighbor pixels. My grid is 128 x 128, which seems small, but that's 16,384 pixels to manage. Each pixel had 8 neighbors so 131,072 checks. 30 times a second. I optimized this with caching so the neighbors aren't computed on every draw.

Next problem was CPU. Even with the caching of the neighbor counts, the game was still doing 16,384 checks 30 times a second. For Pico-8, this was still too much.

There are very efficient algorithms, like Hashlife (https://t.co/9RC7BU7Zf7) that I chose not to implement. Mostly because I don't understand that code and wanted this to be a simple learning exercise for me. So I wanted to optimize, but within reason.

I decided to reduce the grid size to 64 x 64. This changed my computations from 16,384 to 4,096 or a 75% reduction. This had the added effect of making the pixels larger, which I think looked better in the end.

The last and unexpected issue was with the mouse and drawing. I started with a naÃ¯ve solution. When the mouse is down, plot a pixel. Done. Right? Nope. https://t.co/i0ujglUv9r

This naÃ¯ve solution resulted in a disconnected lines. The mouse moves faster than the game loop code. So pixels will get skipped. I think a bit and decide I could cache the previous x,y and with the current x,y draw a line. This would involve a bunch of sin, cos, tan ðŸ˜­ https://t.co/fASYEImFJz

Before spending hours trying to draw a line, I do some googling and find this problem is even more complicated than I thought. Drawing a line might still be too naÃ¯ve a solution. https://t.co/Kw18Gx44qH

This is where I decided to stop. My goal was not to make a drawing app. This part doesn't have to be perfect. It would be nice but I think the holes actually help create the "life". So I'm calling it a feature.

So what is the take-away from all of this? Even with the most simplistic software design you will run into unknown and complicated problems. This is why software estimates are all lies.
