"Why would anybody use vim or emacs when there are modern IDEs around???" Good question! To answer this, I need to know something first: what do we mean by "IDE"? Are we talking about VSCode or IntelliJ? (This thread will be vim-centric btw)

IntelliJ IDEA is a language-specific IDE, designed from ground up for Java and JVM languages. It even has built-in features for Spring and Maven! I have no doubt that a good IDEA user will blow a good vim user out of the water when it comes to JVM programming.

On the other hand, if you need to instead program in Rust, you'll have to learn a different language-specific IDE. So while tools like IDEA are incredible for *specific* languages, they're less suited for *general* programming, which is a constraint a lot of people have.

VSCode, by contrast, is what I'll call a "modular" IDE, or "IDE-lite". It's a set of general-purpose core features with lots of extension points, so that people can write language-specific adapters for the built-in symbol finder/show signature/autocomplete etc.

This provides a much better development experience than unmodified vim. But also this was standardized by Microsoft as the Language Server Protocol, which is designed to work with any editor, so you can use the LSP inside vim, too. Neovim has it built-in! https://t.co/SHc4R7fv5w

Now VSCode was designed ground-up with the LSP, while it's a vim extension (or neovim builtin), so it works a lot more smoothly in Code. I use it regularly for this reason. But even so, it means Code isn't INSURMOUNTABLY better than other editors, which have their own advantages

And what are vim's advantages? I know I'm supposed to start with "modal editing", but I'll skip right past that to by far the biggest: VSCode is designed to be extended by DEVELOPERS. Vim is designed to be extended by USERS.

What do I mean by that? Let's say you wanted your editor to show the wordcount of markdown files. To do that in Code, you'd create an entire "wordcount" extension. The actual code is 100 lines. https://t.co/JYxjSEl2YQ

Now to show the wordcount in vim, just add this to your .vimrc: autocmd BufEnter *.md setlocal statusline=%{%&g:statusline%}\ %{wordcount()['words']} That's fewer characters than the VSCode plugin has LINES!

The difference is that Code assumes the user isn't interested in learning how to extend it themselves and wants to use prepackaged plugins. So it focuses on making plugin use seamless and the plugins play nice with each other, at the cost of making them harder to develop.

Whereas Vim assumes the user will use some plugins, but also wants to be able to directly extend the editor themselves. So everything is exposed to you. On the other hand, it's also easy to fuck everything up, and the onus is on the user to unfuck stuff

For many, many developers, this is a bad tradeoff, since they don't want to dig around in editor guts. I 100% agree that most developers don't need vim, they're better off using a batteries-included editor

But in the right hands, vim's user-centric extensibility is astoundingly powerful. You ever want to compose plugins? You know, run plugin A, run a conditional on the results, and either raise an error or pass something into plugin B? In vim, you can! In vim, it's *trivial*!

Even the developer plugins are designed with user-centric extensibility in mind. Most neovim plugins accept custom callbacks in their config, like with the main fuzzy finder, you can attach custom errors and lookups https://t.co/8GoyIZBOM8

That extensibility is the thing I love most about vim. If I just cared about modal editing or macros I'd just use the vim plugin for vscode (which is also where I recommend beginners interested in vim start) https://t.co/DVtxqujg5s

In short: - You'd use vim over a "true IDE" if you're not working in the blessed set of languages - You'd use vim over an "IDE-lite" if you care more about user-extensibility than a frictionless experience

As a bonus fun thing, here's a script I wrote to set a custom task shortcut, per buffer, for arbitrary commands. Yes, it *can* delete files and stuff, so it'd make a terrible plugin, but it's a great example of the power of user-extensibility. https://t.co/RG956U7qkA
