Our 8 Best Practice Tips for #AWS #KMS 🚦 Least privilege principle 🔑 Key policy 🤝 Permissions 🔐 Encryption 📱 MFA ⌛️ Expiring Grants 🤼‍♂️ Double is trouble 📝 Auditing 🧵👇

🚦 Give only the required permissions to the relevant principals. Do not include kms:* permissions in an #IAM or key policy. Create a top-level KMS policy that denies access to KMS except for the principals that need it.

🔑 Whenever possible, choose to use the key policy over an IAM policy, as it can affect only one KMS key. If you must use an IAM policy, specify KMS keys in the resource element or use values like arn:aws:kms:region:account:key/* or arn:aws:kms:*:account:key/*

🤝 Risky permissions – Pay extra attention when dealing with risky permissions, such as: - kms:PutKeyPolicy - kms:CreateGrant - kms:ScheduleKeyDeletion.

🔐 An #encryption context is an additional layer of #authentication for your KMS API calls. It is a set of key pairs that can contain additional contextual information about the data. The same encryption context is required to decrypt (or decrypt and re-encrypt) the data.

📱Multi-Factor Authentication – You can add #MFA / #2FA for certain API calls to add another layer of security. You can add a condition to the key policy that checks for when or if an MFA device was used as part of authentication.

⌛️ Grants do not expire on their own – Remember to retire or revoke them when they are not needed anymore.

🤼‍♂️ Delete duplicate grants that have the same: - Key ARN - API actions - Grantee principal - Encryption context - Name. Duplicate grants can lead to unintended privileges if one is retired or revoked.

📝 Auditing - Enable #CloudTrail logging to ensure that all KMS API calls made on keys in your AWS account are automatically logged. Use this information to determine which request was made, the source IP address from which the request was made, who made the request, and more!
