6:52 am, a founder gets a call: they have a $65k AWS bill (their normal is $10k). What's the lesson here? (spoiler: it wasn't a hack) ðŸ§µ

From a non-tech perspective, the lesson is that it's not game over. Things happen all the time, and you just deal with it, like it or not.

In this case the founder talked to AWS and they reduced the bill to $26k. They also helped the founder fix the issue, bringing the next bill down to their regular $10k.

From a tech perspective there's a different lesson. You can't expect your software and infrastructure to work all the time. You need to: - understand how it fails - know when (not if) it's failing You need visibility, and you need alerts.

This situation happened because a scaling process threw an error and didn't clean up. Alerts on that error could have prevented this. Also, an AWS Budget could have alerted the founder about unusual costs.

They did have alerts! But not for that error. Who would expect an error on a scaling process? That's why you need to understand how things fail. If you don't understand how it can fail, you don't know what to monitor.

Security is another example (even though this wasn't a hack). You can (and definitely should!) apply all the best security practices, such as MFA. But you're not invulnerable, and if something happens, you need to know.

Security folks call it: - Preventive controls: Strengthen the system against incidents. - Detective controls: Detect and react

Wait, can testing help me catch this? Yes, but no. Testing specifies behavior for known data, monitoring confirms behavior in the face of unknown data. You need to do both.

(Recap) Lessons learned: - Non-tech: It's not game over. Deal with the problems. - Tech: Understand how things fail, and monitor for failure (set up alerts).

Good monitoring is hard. Google's SRE book has a lot to say about it. But the basic stuff like billing alerts is super easy.

PS: Setting up billing alerts in AWS takes 15 minutes. Let me know if you need help with that.
