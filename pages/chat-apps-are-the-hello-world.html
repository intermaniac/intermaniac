Chat apps are the 'hello world' of real-time data. Opening up a socket connection and 2 browser tabs is still feels magical to me üòÖ Coupling this with GraphQL is honestly one of my favorite parts of modern development. But most chat apps won't scale. AWS AppSync changes that üßµ

By defining a schema, we essentially define all of the operations our API supports. A lot of GraphQL runtimes handle Queries and Mutations. AppSync takes it further by offering subscriptions. The `@aws_subscribe` directive simply maps to a mutation. https://t.co/DTgS4BkiHQ

Ok--we defined what we want to do, but how do we define where the data is stored? Even better--how do we do it in an automated way, not just by clicking through a GUI? The AWS CDK. Use TypeScript to define your backend services, like this database https://t.co/MJm2fJQeAX

So we know our operations, and we know where our data is being stored. But how do we connect the two? Most GraphQL providers use a JavaScript function to _resolve_ your data. Hence the name "resolver". You can do that in AppSync too, similar to adding this User to our DB https://t.co/NVUAFFz7va

The problem here is that well...the function isn't really doing anything. It's just passing data. On top of that, we now have to manage this code. AppSync can directly tie into our DB w/o using a function you have to manage! https://t.co/WDvdkM2ASs

Ok ok, that .vtl file makes things super fast, but it's kinda gross right? I think it'd be cool if we could write in JavaScript/TypeScript and AWS would just turn into that VTL stuff w/o me having to think about it. If you agree, know we're listening üòâ https://t.co/ILi4QWy5FO
