An under-appreciated skill to have an an analyst is a decent understanding of the powers of a CI/CD platform and the ability to work with GitHub Actions. If this is new to you then I hope this thread helps you go from zero to one (1/ðŸ§µ)

CI/CD is a methodology that makes it easier to ship products more frequently by having automations in place to take care of builds, tests, and deployments. Devs have been using these practices for a while, however, I haven't seen it catch on as much among #AnalyticsEngineers.

GitHub Actions is one of the most popular CI/CD platforms and for many people the obvious choice to get started with bringing tried and trusted DevOps principles into their data teams.

The learning curve is quite modest and all you need is an intermediate knowledge of YAML, Bash (command line), Environment Variables, and of course GitHub. Fortunately, the proliferation of @getdbt has made a lot of us comfortable with these pre-requisites.

In a nutshell GitHub Actions are a set of IFTTT like rules that can create automations (called jobs) for myriad tasks. A great place to get more acquainted with GitHub Actions is on their docs https://t.co/EHenMeQU6H

One such task is automatically creating PRs. If you work with a preprod and production environments for your data models then you could set up a simple action that creates a PR on production every time something is merged into preprod.

Another simple job could be to run and test all your dbt models as part of your PR and then set rules that allows the PR to be merged only if all tests have passed.

A more exciting use case is running siloed python scripts; e.g. a regression tester that runs queries to check if things still add up after making changes to your data model. You can run such python script automatically and make it fit into your dev workflow seamlessly.

The best part is that none of these are theoretical examples. I have personally deployed these into my team's workflow just over the past week having had very little exposure to GitHub Actions before that.

There are so many more use cases I can think of. Of course, this depends on your workflow and setup, but I'm sure there's a lot of value that us data folks can unlock with a little bit of GitHub Actions knowledge.
