SQL (Structured Query Language) is very popular and rightfully so - it is essentially a data manipulation language! but it isn't just one language - it is a type of languages. We already use structured data in our contracts (ABIs), in web3 API (JSON-RPC) and all over web2 tech.

And anything that has a well defined structure can be queried and manipulated with SQL: JSON CSV Protobuf TypeScript definitions YAML Machine Learning Models

Now imagine you want to come up with some useful query *after* the contract was deployed: You would have to rewrite the contract, do an audit and deploy and upgrade - which is a risk within itself. And what if a user comes up with a useful query that you didn't come up with?

Take for example Dune Analytics - the users are ones who create all sorts of amazing queries the creators couldn't even think of. But all that happens *off-chain*. Is there a case for SQL *on-chain*?

1. Blockchain = Database But even now after years of R&D it doesn't feel that way. About 3 months ago I had this thought: wouldn't it be cool if we could get the classic databases experience for blockchains?

2. Transaction = Query (also known as... Transaction) Wikipedia: "A transaction generally represents any change in a database." Blockchains already behave like databases, because...

3. Blockchain is an innovation of databases. Blockchain is a special type of Database that stores its own transactions in an append-only chain - a Blockchain.

This innovation together with innovations in consensus algorithms brought us the Trustless Blockchain we know and love today. But in order to scale early blockchains, efficient execution environments were implemented, such that we lost much of the Database component in them.

As scaling technologies develop rapidly (and faster than anticipated) the potential for SQL blockchains skyrocketed to over 9000! In particular, Zero Knowledge technologies can enable SQL-powered blockchains at fraction of the cost of a regular L1 today!

Now imagine: What if SQL wasn't just implemented in EVM... What if SQL *was* the VM! We can create a blockchain whose execution engine is SQL!

The experience of classic databases with all the familiar querying languages, tools and infrastructure... ...such that any shared database today can be onboarded to Web3 and start benefiting from all of blockchain's properties such as security, finance, etc. Just to name a few:

The implications could be MASSIVE. Google search index as a decentralized, transparent database. Artificial Intelligence models proved on the blockchain. Software supply chains integrated with DeFi, staking, bug bounties and treasuries. Twitter with privacy & accountability.

So why SQL GUD? By bridging the gap between Web2 <-> Web3 technologies we can on-board the entire existing Internet infrastructure. Because if we can't bring the Internet to our blockchains... We should bring new blockchains to the Internet.
