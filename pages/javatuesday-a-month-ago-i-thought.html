#JavaTuesdayThread A month ago I thought that building #Java libraries out of C++ code using Project Panama is all about two things: 1. Create a C struct that represents the C++ class. 2. Creating proxy methods against that C struct. How wrong I was [thread]…

Most modern C++ libs were developed in C in the past, which means they had structs, functions, globals, etc. But obviously, the only way to evolve was to move to C++ which is much better in terms of encapsulation, templating, etc.

But now, in JDK 19 we’ll get Foreign Function & Memory API in their first preview - an implementation of the ABI. The only downside it can work with C native code which good thing at the very beginning.

Some time ago, I did a few projects whose purpose was to degrade C++ API to C API in order to integrate C code into CPython. And now, the same approach is possible for Java.

So, no matter what kind of C++ variable you’re working with, just keep in mind that it’s only a memory segment or a pointer. The fact that it has a type is secondary. Let me explain it.

C++ variable type matters only to you as a developer (and a compiler) but not to the runtime. For instance, if you have a C++ vector instance, then nothing stopping you from creating an un-associated pointer to that vector that can be processed within C code or through the ABI https://t.co/74UGWmlKvi

A newly constructed un-associated pointer is a pure piece of memory that can be consumed by any higher-level system like JVM. So, if you ever wanted to have the functionality of C++ vector but in Java, you surely can get it within a few lines of code: https://t.co/vTxgkGTka6

Such an approach makes interaction with the C++ libraries as smooth as possible. So, one day after the JDK 19 release you already have an answer to a question about how to work with something beyond C because it’s old and ugly.

Even more, a few other alternatives to an-associated pointers are far more work. The first one - is C++ to WebAssembly to C to Java. I don’t even want to dig into that, it just doesn’t work nor will it be a conscious developer choice.

the alternative is better but bound to the trade-offs - degrade C++ API to C. I tried a few times. It didn’t work well, because it’s not just turning a C++ class into a C struct to make it work in Java, it’s a choice to degrade some fine-grained APIs to nothing.

The only common ground among these approaches is a need to create an intermediate entity or a proxy that knows how to restore the original type and C function for each of the C++ class methods, which is inevitable C++ native symbols unavailable through C ABI. https://t.co/HpxVcbitN5

Anyways, we’re one week before the JDK 19 release (imagine that!) and still have a lot to talk about. Project Panama is like a Klondike, it’s full of features and it’s really nice technology to work with. #Java #OpenJDK #JDK_19
