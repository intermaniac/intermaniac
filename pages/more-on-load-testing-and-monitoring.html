More on load testing and monitoring load testing targets in general! Alongside my previous ðŸ§µ, we want to set baseline system performance thresholds for the application host. #devops #SoftwareEngineering #codinglife #aws #azure #gke

This can help us detect things like memory leaks or regression in cpu performance. It could even go as far as looking into things like disk or network io, but all of these depend heavily upon the underlying hardware and infrastructure

Be sure to run the exact same tests against your app while hosted on different hardware you might expect customers to use. For example, ec2 instance types in aws. There are SO many variables to consider in a deployment, so scope down to as many (or even 1) reliable deployment

I want to capture these system metrics in CI during load tests. One approach was InfluxDB + Telegraf. This uses a push approach. The Telegraf agent is installed on each node, and configured to record metrics via plugins (there are MANY!) and push into InfluxDB

The Flux query language and docs are wonderful! Functional, easy to read! There is infra and config overhead for deploying this for one-time use throughout the ci build. I figure we can dump this db as an artifact for realtime analysis of tests for teams offline

It would be easy to write queries to determine if metrics underperformed under some arbitrary threshold (i.e. mem usage increased and is never freed)

To reduce infra overhead from standing up another node in an already complex environment, we can instead deploy prometheus exporters on targets. The CI agent can deploy prometheus locally and be configured to scrape metrics from targets. Use PromQL in place of Flux. Same outcome

Ofc, the agent would also need a route to each target. #devsecops is important! Public facing nodes are possible, ser aws managed IP lists, security groups, etc. Or deploy your CI agents on the same network / connect them via VPN/IPSEC

Clearly, management of these systems and networks can spiral into very complex situations! I also don't recommend running load tests across the public network ðŸ˜‰, probably better to use a jump host deployed in your public subnet that has a local route to the targets instead

LMK if you have any other suggestions on automating the capturing of system metrics of targets of load tests in CI!
