1 - Inspect the hottest paths in your app The best way to avoid over-optimization and add value to an application while improving performance is to work where customers more often access your application. /2

NPM packages like clinic, 0x, and Google Tracing being executed with autocannon (or other load testing tools) would help you a lot to find blind spots that could be refactored and change significantly your app response time. /3

2 - Keep Node.js and packages up-to-date After inspecting and looking for performance issues you'll notice that most of the problems are caused by 3rd-party packages thus you don't have access to change yourself. /4

Hopefully, when people notice some critical problems in NPM packages, those packages release security and performance patches that fix problems as fast as they can. /5

However, is not so common for people to keep updating packages weekly or monthly thus makes them vulnerable and slow until they install updates /6

3 - Use an Application Performance Monitoring to get to know issues before users do Once an application is live anything can happen. Frequently errors, slow down in specific periods, application crashes, and more would be good reasons why users give up using your product ðŸ˜± /7

Adding alerts and monitoring how an application is handling accesses and how it is fault tolerant would leverage you to fix issues before your users notice them. /8

4 - Improve legacy code + Run benchmarks! We all should have a routine for improving existing legacy code that our users access most often. The best way to check if a refactoring was more valuable than the existing code is to measure it by running benchmarks /9

There're projects that add a routine to compare how was the performance before and after a new version is released so they can understand how the software's lifetime value is improving. /10
