"That was some serious AWS kung fu" Those were the words of the CTO of a startup for which I was contracting. Story time! ðŸ§µ

Early Tuesday morning, a cup coffee in my left hand, I tried to log in to the company's AWS account. Woops, login error! Ok, try again. Nope, login error. I sip my coffee trying to wake up, and I wonder... could it be? Nah, it's just my user.

I ping the backend dev on Slack and ask him to log in. He can't either. "Are you sure, Lio?" (yup, same first name as Messi). "Yeah, 'Your authentication information is incorrect.'" Trust me, I was wide awake at this point.

I was about to sound every alarm, but I remembered the "devops" IAM user used by the GitLab CI/CD. Let's give it a shot! I got the keys from GitLab, set up the CLI and crossed my fingers. It worked!

Using some super secret CLI mumbo jumbo (i.e. running the aws iam list-users command) I found that our IAM users had been deleted!

Incident Response mode activated! - I set up a war room - I pinged the CTO - I pinged our PM and told him to ping the CTO - I sent an @ here on Slack - I checked CloudTrail

Who deleted our users? THE ROOT USER!! CTO: "Can't we delete the root user?" Me: "No, we can't. But perhaps... Can you create a new AWS account and give me access?"

Did you guess my kung fu? Here it is: - I set up an Organization in the new account - I set up a Service Control Policy to restrict all actions from the root user - I invited the old account, and accepted the invite from the "devops" user "That was some serious AWS kung fu."

PS: Nope, we couldn't just change the root user's password. The account had been created by the dev agency formerly contracted for the project. And they were the ones attacking. Lesson: Don't let a dev agency own YOUR resources.
