Ok!! Time to get this puppy rolling. The Goal: git gud at #BugBounty The Method: Labbing, lots and lots of Labbing. This thread will be my write up of what I learned today. Disclaimer: IDK what I'm doing right now, don't take my word as authoritative. On Anything. Ever.

Today's labs were focused on SQLi and XSS. Pretty neat stuff, I had an introduction to SQLi through an old Computerphile video ages ago, so the beginner labs on Portswigger Academy were pretty quick to clear. But what is SQLi you ask? Here's my attempt to explain...

SQLi is short for SQL injection, the bare bones basics is when an app takes user input and puts it in a SQL query. Pretty common use case would be when implementing a search function.

It goes something like this: User enters a search keyword, server receives the search and places it directly into a SQL query to the database, the database returns the data from the query which is then sent back to the user and displayed as their search results.

Now, lets say I a real mean guy, and I decide to sent something else as my "search" maybe is sent a ' -- instead. Turns out that those characters have special meaning in SQL, which means I can modify the query to return data I'm not supposed to have.

Naturally, SQLi has been a known vuln for a hot minute, so it usually won't be that simple. But the basic takeaway is to pay attention when user input is being used to perform server side functions. If that input isn't checked properly, I can get your server to give me things.

Onwards to XSS, which is an abbreviation for Cross Site Scripting. XSS comes in a few general flavors - Reflected - Stored - Dom Based NGL, The XSS stuff still makes my brain hurt, but I think I got the general idea...

Broadly speaking, XSS revolves around getting custom (often malicious) Javascript to be run by the browser. In the case of reflected XSS the bad guy JS will be make use of the web server inserting user input into http responses. A good example is our old friend: Search

Say I make a search on some blog or news site, usually, when the search results load, I get anew page that has my search term displayed, something like "You Searched for: Hackin" Now, what can happen, is if i make my search a Malicious <script> tag...

That tag will be taken by the server and inserted into the html of the response. Meaning I can get the resulting page to load with my own, custom JS. Why is this bad? Well, since that input might be passed using the URL, I can create a Evil URL and sent it to people...

If those people click the link, it loads that page up with my evil JS, which allows me to do loads of wicked things, like steal their delicious cookies. Stored XSS operates simularly. The primary difference is instead of using a URL, I can get my JS to be stored by the DataBase

I can pull that off through something like a comment or tweet, that gets stored as part of the site. The end result being that any page that causes that data to be loaded and used as part of the site can be contaminated with my Evil JS.

Ok, Finally, the Dom Based XSS. This one still hurts my head, but I'll give it a go. the gist of the scene seems to be using the actual code of the webpage itself. Often, websites make use of functions that directly modify the page without reloading it...

In case where it take user input to do so, there are options to get Evil JS into the page once again. Honestly, thats about the best I can do with it right now. understanding DOM manipulation with JS and JQuery are definitely added to my list of things to research.

All things considered, The key takeaways from today for Bug Hunting seems to boil down to this: Understand how the App takes and uses User Input, if the app isn't filtering and controlling it properly, you can probably breach data or construct Evil Webpages.

Also, I really need to get familiar with JS analysis tools, trying to do some of those labs with pure page source reading was a pain. Until next time, Happy Hackin'
