Item 9: Prefer try-with-resources to try-finally üßµ‚¨áÔ∏è #Java #LearnJava #JavaDeveloper #Programming #Coding

Many Java libraries include resources that need to be closed manually through a close() method. E.g., InputStream. If you forget to close resources it can be a major performance killer!

The problem with try-finally is that is it can make debugging much harder as the root exception (the one you most likely care about) gets suppressed. In the example below if readLine() throws and then close() also throws, the original exception is lost! üî®

https://t.co/Aoboaqc6Yv

This is solved by try-with-resources! To be used in this way the resource must implement the AutoCloseable interface. As you can see in the example below it is much more concise than try-finally.

https://t.co/aoPEYkK17L

In this case if readLine() and close() both throw the first exception is preserved with later one suppressed. But this does not mean it is lost! It is printed in the stack trace but annotated as suppressed and can be accessed programatically with the getSuppressedMethod()
