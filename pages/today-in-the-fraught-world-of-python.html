Today in the fraught world of Python packaging, this is an interesting article that misses some important technical details but really captures an emotional reality of the “python” brand in certain spaces. Here is a meandering thread in response: https://t.co/60rdUVJSJm

You may notice that none of the 4 examples of Python programs that I'm confident actually *work* are *not* things that you `pip install`. Python is an implementation detail. You get them through whatever mechanism you otherwise get similar applications or systems.

In other words, the platform is “ubuntu” or “macOS” or “Windows”, not “go to python dot org or maybe pyenv or maybe asdf, or you know what actually maybe you want to use the python3 that ships with xcode, or homebrew, it's really up to you”

You may also notice that none of these things use the preferred solution of this genre of critique, which is “single binary”. Nobody actually wants a single binary. Applications don't ship that way in real life.

For example, consider a big, successful Go application, that “just works”: Docker. The “single binary” I have installed for Docker on my mac consists of more than three thousand files in a .app bundle. And, oops, it's got >150 binaries and … >50 Python scripts in it, too.

*Sometimes* a single binary can make managing this process easier, but what (a certain subset of) people want is `(brew|apt|nuget|port|mas) install`.

The idea that you should rewrite your program in Go in order to address dependency management is the idea that you should drive your car off a cliff if its muffler is making a weird noise. Very silly. Don't do that.

My favorite existence proof of "switching to Go doesn't solve the problem" is this case study: https://t.co/djQkfJ6XaD . It seems to still be on Python 6 years later, so I guess it worked out. (The point isn't "Go sucks" but rather "using Go doesn't solve this specific issue.)

On the other hand, I am not defending the Python ecosystem status quo either: if your car's making weird noises, maybe you should actually get it fixed before it explodes while you're driving.

Shipping a Python app as a PyPI package introduces a plethora of failure modes that most users don't need to know about. If you tell them to literally `pip install`, then you're installing into some shared namespace that will break their other Python applications.

Even if you tell them to `pipx install` (a huge improvement!), you're still introducing a bunch of weird edge cases where they need to know which python interpreter they're using and whether or not it's going to break when they upgrade.

PyPI is not useless as a deployment target. It's often *my* preference, because I want to actually poke at the code for these things, and I know how to manage my venvs and trivially fix any issues that may arise, so I'm willing to pay that cost.

But when I get software in Ruby, I don't want a `gem install`, I want to `brew install` or `apt install` and forget about it.

Side note: as a non-Ruby developer, I sometimes have the "python" packaging experience with Ruby tools that are similarly uploaded to rubygems and not packaged for end-user consumption. Yet, homebrew itself is in ruby and I've never once seen a ruby-specific failure mode from it.

We sometimes think of ourselves as "Python" developers, but if we want to reach users on Windows, macOS and Linux, then we are Windows, macOS, and Linux developers.

Different audiences are different, though. You don't necessarily need to submit something to an app store if your audience is "developers, but not Python developers". These paths are not as easy as `twine upload`, but they're not necessarily super hard, either.

For example, if you are closely connected to the area of the social graph I inhabit and you want to distribute a command-line tool to developers on macOS, consider reading https://t.co/aRTPdjZqz5

On Ubuntu-like systems, a similar option might be https://t.co/MjF3bw7L0Z (and https://t.co/1ZS34oahdY)

I honestly don't know what the best option is for Windows, since I don't use the platform enough, but I suspect there's something relatively low-friction there too.

A thread like this one isn't a great resource, and obviously lacking something major like Windows and Redhat platforms means it's woefully incomplete. We probably need to update https://t.co/0F3p7NAsoX to include two very distinct paths: "for Python developers" and "for users".
